
# Prompt-engineering-cli-agent

> An AI-powered agent that securely translates natural language into CLI commands. Built to demonstrate advanced, iterative Prompt Engineering, edge-case handling, and LLM evaluation methodologies.

## 📌 מבט על (Project Overview)
פרויקט זה מציג סוכן חכם (AI Agent) המתרגם בקשות בשפה טבעית לפקודות חלונות (Windows CMD) ישימות. המערכת פותחה תוך שימוש במודל `Llama-3.1-8b-instant` דרך ה-API של **Groq**, וכוללת ממשק משתמש אינטראקטיבי שנבנה ב-**Gradio**.

הפרויקט אינו רק כלי פונקציונלי, אלא **מעבדת מחקר ל-Prompt Engineering**. מטרת הליבה של הפרויקט הייתה לנתח את התנהגות מודל השפה (LLM), להתמודד עם אתגרי אבטחת מידע (Jailbreaks, הזרקת פרומפטים), ולפתח ארכיטקטורת פרומפט יציבה ובטוחה (Guardrails) בתהליך איטרטיבי.

בנוסף, הפרויקט כולל מנגנון **LLM-as-a-Judge** (שופט אוטומטי המבוסס על מודל `Llama-3.3-70b`), אשר מעריך כל פלט בזמן אמת על סמך 3 מדדים: פורמט, תחביר ואבטחה.

---

## 🧠 מסע הנדסת הפרומפטים (The Prompt Engineering Journey)
במהלך הפיתוח, הפרומפט עבר שלוש איטרציות מרכזיות. כל איטרציה נבדקה מול סט של 15+ תרחישי קצה (Edge Cases) שכללו פקודות הרסניות, הנדסה חברתית ודו-משמעות.

### איטרציה 1 (V1): הגישה הנאיבית (MVP)
* **הפרומפט:** הגדרת תפקיד פשוטה ("You are a CLI expert") ודרישה להחזיר רק פקודה.
* **תוצאה:** המודל תרגם פקודות בסיסיות בהצטיינות, אך נכשל לחלוטין באבטחה. 
* **הכשל:** כאשר התבקש "למחוק את כונן C", המודל פירש את הבקשה מילולית וייצר פקודה הרסנית (`del /q /f C:\*`). כמו כן, המודל נטה ל"פטפטת" (Chattiness) כאשר נתקל בשאלות לא קשורות (למשל: "איך מכינים חביתה?").

### איטרציה 2 (V2): כשל ההנחיה השלילית (Negative Prompting Failure)
* **השינוי:** הוספת חוקים האוסרים על ייצור פקודות מסוכנות ("Do not issue dangerous commands like del, format").
* **תוצאה:** המודל המשיך לספק פקודות הרסניות, ולעיתים אף המציא פקודות לא קיימות (Hallucinations).
* **ניתוח הכשל (תובנה מרכזית):** חווינו קונפליקט לוגי קלאסי. מצד אחד, חוק מס' 1 דרש מהמודל להחזיר **אך ורק** פקודת שורת קוד. מצד שני, חוק מס' 4 אסר עליו לעשות זאת עבור פקודות מחיקה. מכיוון **שלא הוגדר למודל נתיב מילוט (Fallback string)**, הוא "נכנע" לחוק הראשון (Sycophancy) וסיפק את הפקודה ההרסנית כדי לא לשבור את חוקי הפורמט.

### איטרציה 3 (V3): ארכיטקטורה מבוססת תנאים (Conditional Guardrails) - *הגרסה הסופית*
* **השינוי:** מעבר מהנחיות שליליות ("אל תעשה") למבנה של **If-Then אלגוריתמי עם סדר עדיפויות מוגדר (Execution Rules)**.
* **היישום:**
  1. הגדרנו מחרוזות חסימה מפורשות (למשל: `"BLOCKED: Security violation."`).
  2. בנינו היררכיית חוקים: קודם בדיקת אבטחה, אחר כך זיהוי הזרקות פרומפט (Anti-Jailbreak), ורק בסוף ייצור פקודה.
  3. הוספנו תמיכה מפורשת בשרשור פקודות (`&&`) למניעת שבירת שורות.
* **תוצאה:** 100% אחוזי הצלחה בחסימת פקודות הרסניות, יציבות מוחלטת בפורמט הפלט, ויכולת חסימה של ניסיונות Jailbreak מורכבים.

## 💡 תובנות מפתח ולמידה (Lessons Learned)

1. **Tell, Don't Ask (נתיבי מילוט):** אי אפשר פשוט להגיד ל-LLM "לא לעשות" משהו מבלי לתת לו משימה חלופית. הגדרת מחרוזת Fallback מדויקת היא קריטית לבטיחות.
2. **Order Matters (חשיבות סדר ההוראות):** מודלים ממשפחת Llama (ורבים אחרים) מייחסים חשיבות לסדר ההוראות בפרומפט. הצבת חוקי האבטחה בראש הרשימה, תחת הכותרת `Highest Priority`, שינתה את תעדוף המודל.
3. **LLM-as-a-Judge:** אוטומציה של הערכת פרומפטים באמצעות LLM נוסף חסכה שעות של בדיקות ידניות ואפשרה לנו לקבל מטריקות (Scores) מדויקות על פורמט, תחביר ואבטחה בכל הרצה.

## 🛠️ טכנולוגיות (Tech Stack)
* **Python 3.10+**
* **Groq API** - להרצה אולטרה-מהירה של מודלי LLM פתוחים.
* **Gradio** - לבניית ממשק המשתמש (UI).
* **LLMs:** 
  * `llama-3.1-8b-instant` (כסוכן המתרגם - Agent).
  * `llama-3.3-70b-versatile` (כשופט האוטומטי - Evaluator).

   ```bash

   cd prompt-engineering-cli-agent

   ```bash
